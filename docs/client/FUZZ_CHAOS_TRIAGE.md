# Fuzz + Chaos Triage Policy

## Scope

This policy governs nightly fuzz/chaos evidence generated by:

- `.github/workflows/nightly-fuzz-chaos.yml`
- `tests/tooling/scripts/ci/nightly_fuzz_chaos_report.mjs`
- `core/local/artifacts/nightly_fuzz_chaos_report_latest.json`

## Cadence

- Nightly on GitHub Actions schedule (`cron: 13 5 * * *`)
- On-demand via workflow dispatch
- Local reproduction:
  - `node tests/tooling/scripts/ci/nightly_fuzz_chaos_report.mjs`

## Severity Mapping

- `S0` Critical:
  - `enterprise_hardening_strict` fails
  - deterministic safety-plane contract failure in report payload
- `S1` High:
  - `formal_invariants` fails
  - repeated nightly failure in same check for 2+ runs
- `S2` Medium:
  - fuzz preflight/target-list errors with remaining checks passing
- `S3` Low:
  - transient infra or environment noise without reproducible failure

## Response SLA

- `S0`: immediate stop-ship for affected lane, triage started same day
- `S1`: triage within 24 hours
- `S2`: triage within 3 business days
- `S3`: queue in regular maintenance cadence

## Required Triage Record

For every failed report, record:

- artifact path and run identifier
- failed check IDs
- reproduction command
- owner
- disposition (`fixed`, `accepted-risk`, `false-positive`)
- linked follow-up issue/PR

## Closure Criteria

A failed nightly cycle is considered closed when:

1. failure is reproduced or explicitly classified as infra-noise,
2. fix is merged (or risk accepted with rationale),
3. a subsequent nightly run returns `ok: true`.

Purpose: raise Protheus to Fortune 100 operating standards without shrinking the capability graph or weakening the portable layered architecture.

## North Star

Protheus should behave like an AI-native operating system substrate:

- Rust remains the kernel and portability anchor.
- Conduit remains the only sanctioned bridge into flexible TS surfaces.
- Capabilities stay broad, but supportability becomes explicit by profile, layer, maturity, and owner.
- Every supported feature must have a contract, owner, test path, packaging path, and rollback path.

## Standards Target

This program is complete when all of the following are true:

1. `npm run test`, `npm run test:ci`, `npm run test:vitest`, `npm run js:holdout:audit`, and `npm run ops:ci-quality:check` are green together.
2. A normal operator workflow does not dirty tracked files outside approved generated artifacts.
3. Every command and major capability maps to one manifest with `owner`, `maturity`, `layer_floor`, `profiles`, `risk_tier`, `state_writes`, and `tests`.
4. Production entrypoints run from deterministic dist artifacts or Rust binaries, not ad hoc source bootstrap paths.
5. Supported runtime profiles are generated from layer-aware manifests rather than tribal knowledge.

## Workstreams

## 12-Week Backlog

### Weeks 1-2: Stop contract drift

1. Align wrapper-contract policy, tests, and implementation so conduit-wrapper modules have one enforced shape.
2. Make the CI harness and local test commands agree on the same release truth.
3. Clear strict JS holdout failures and classify every remaining `.js` file as `runtime_bridge`, `compat`, or `migration debt`.
4. Add one red/green dashboard document for release truth instead of relying on scattered command outputs.

### Weeks 3-4: Introduce capability manifests

1. Add a capability manifest schema and registry generator patterned after the command registry.
2. Generate operator docs, security inventories, and profile support matrices from manifests.
3. Add maturity classes: `ga`, `beta`, `experimental`, `compat`, `retired`.
4. Add explicit `layer_floor` and `profile_allowlist` fields for every capability.

### Weeks 5-6: Make layers and profiles the product boundary

1. Define supported profiles: `seed`, `edge`, `desktop`, `cluster`, `embedded`.
2. Map `kernel_layers` features to those profiles and generate a profile matrix artifact.
3. Require every capability to declare runtime budgets and unsupported profiles.
4. Add profile boot smoke tests and packaging checks.

### Weeks 7-8: Enforce coherent module archetypes

1. Define allowed module archetypes: `primitive`, `controller`, `store`, `adapter`, `wrapper`, `tool`, `compat`, `registry`.
2. Add guardrails for allowed imports, allowed state writes, and required exports per archetype.
3. Move compatibility logic out of ambiguous runtime lanes and into explicit compat wrappers.
4. Reduce accidental direct file-system mutation from general `client/systems/*` modules.

### Weeks 9-10: Clean runtime/output boundaries

1. Move all runtime-emitted files out of tracked source paths unless explicitly generated by sync commands.
2. Separate generated docs from runtime state and require explicit regeneration commands.
3. Make `dist` vs `source` mode a hard environment split with the production path favoring dist only.
4. Add a repo-cleanliness gate for tracked mutable artifacts.

### Weeks 11-12: Enterprise operating model

1. Expand ownership beyond broad top-level paths.
2. Add SLOs for release truth, CI stability, packaging success, and profile boot success.
3. Add release rings and rollback policy for `ga` surfaces.
4. Freeze for a hardening pass, then run gameday, DR, and install-path validation on each supported profile.

## Sequencing Rules

1. Do not start capability-manifest work until release truth is green.
2. Do not tighten SLO thresholds until CI noise is under control.
3. Do not move modules between archetypes without updating the manifest and the guard policy in the same change.
4. Do not remove compatibility layers until profile packaging and rollback paths exist.

## Weekly Success Metrics

1. Number of red release-truth gates
2. Number of uncategorized capabilities
3. Number of uncategorized JS holdouts
4. Number of tracked runtime-mutated files
5. Number of supported profiles with green boot smoke
6. Mean time to explain ownership/support status for any command or capability

## Desired End State

Protheus remains broad, ambitious, and portable, but it stops behaving like a growing research tree and starts behaving like a productized substrate:

- capabilities preserved
- support boundaries explicit
- profiles first-class
- contracts generated from one truth
- runtime outputs isolated
- enterprise confidence earned by repeatable evidence
